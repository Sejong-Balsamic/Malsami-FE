# 진짜 반응형 전환 — 통(640/960 컨테이너) 완전 제거 설계

날짜: 2026-06-18
배경 이슈: #655 후속 "반응형 전환" (시각 검증 필요로 분리됐던 항목)
선행 작업: 컨테이너 폭 토큰화 + 공용 레이아웃 컴포넌트 도입(커밋 2186774, 미배포)

## 문제 정의

현재 앱은 `ClientLayout`이 화면 중앙에 **고정 폭 통**(모바일 640px / PC 960px)을
하나 두고, 모든 페이지·헤더·네비를 그 통 안에 가둔다. 이 구조에서는 PC에서
"모바일 화면을 가운데 띄운" 인상을 벗어날 수 없다. 콘텐츠를 아무리 손봐도
통이 폭을 막기 때문이다.

## 목표

통을 **완전히 제거**하고, 화면 폭에 따라 레이아웃이 재배치되는 진짜 반응형으로
전환한다. (그래폰/토스 스타일)

- **모바일(기본)**: 지금처럼 1열 풀폭 + 하단 탭바(CommonNav). 시각적으로 변화 없음.
- **PC(lg 1024px+)**: 상단 글로벌 가로 네비 + 넓은 중앙 콘텐츠 영역(그리드 활용).

## 골격 결정 (확정)

- 레이아웃 구조: **통 제거 · 진짜 반응형**
- PC 골격: **상단 글로벌 네비 + 넓은 콘텐츠** (사이드바 아님)
- breakpoint: 2단계 (모바일=기본 / PC=lg). md/sm 신규 도입 안 함.

## 핵심 설계

### 1. ClientLayout 재구성 (통 제거)

현재:
```
<div flex justify-center bg-gray-100>
  <div max-w-container lg:max-w-container-lg bg-white>  ← 통
    {children}
  </div>
  {CommonNav}  ← 모바일 하단 탭바
</div>
```

변경 후:
```
<div min-h-screen bg-white>
  {/* PC 전역 상단 네비 — lg에서만 보임 */}
  <GlobalTopNav className="hidden lg:flex" />

  {/* 콘텐츠 영역 — 폭 제한은 페이지/섹션이 콘텐츠 성격에 맞게 자체 결정 */}
  {children}

  {/* 모바일 하단 탭바 — lg에서 숨김 */}
  {shouldShowNav && <CommonNav className="lg:hidden" />}
</div>
```

- 바깥 회색 배경(`bg-gray-100`) + 중앙 흰 통 제거 → 전체가 흰 배경.
- 폭 제한을 ClientLayout이 강제하지 않는다. 대신 **콘텐츠 폭은 각 화면이
  "읽기 콘텐츠는 좁게, 그리드 콘텐츠는 넓게" 자체 결정**한다. 이를 위해
  콘텐츠 폭 토큰/유틸을 재정의한다(아래 3).

### 2. 네비게이션 이원화

- **모바일**: 기존 `CommonNav`(하단 fixed 탭바) 유지, `lg:hidden`.
- **PC**: 새 `GlobalTopNav` — 상단 가로 네비(로고 + 자료/질문/마이 메뉴 + 검색/알림).
  `hidden lg:flex`. 기존 헤더(`Header`/`LandingHeader`/`CommonHeader`)와 역할이
  겹치므로, PC에서는 페이지별 fixed 헤더를 숨기고 GlobalTopNav가 대신한다.
  (모바일은 기존 페이지별 헤더 유지.)

### 3. 콘텐츠 폭 토큰 재정의

통이 사라지면 "모든 콘텐츠 640/960" 단일 폭은 부적절하다. 콘텐츠 성격별로 나눈다:

- `max-w-content-narrow` (예: 본문 읽기용, ~720px): 상세 본문, 글쓰기 폼 등
  "한 줄이 길면 읽기 나쁜" 콘텐츠.
- `max-w-content-wide` (예: ~1200px): 게시판 리스트/그리드, 홈 등 "넓게 펼칠수록
  좋은" 콘텐츠. 중앙 정렬.
- 모바일에서는 둘 다 `w-full`처럼 동작(화면이 토큰보다 좁으므로 자연히 풀폭).

tailwind.config의 maxWidth에 위 토큰 추가. 기존 `container`/`container-lg`는
점진적으로 신규 토큰으로 대체(또는 wide=960 유지 후 확장).

### 4. fixed 요소 정렬 기준 변경

현재 헤더/네비/토스트/하단바는 `max-w-container`(통 폭)에 맞춰 중앙 정렬돼 있다.
통 제거 후에는:
- 모바일: 풀폭 fixed (지금과 동일하게 보임).
- PC: GlobalTopNav가 상단을 담당하므로 페이지별 fixed 헤더는 `lg:hidden`.
  하단바(댓글 입력 등)는 PC에서 콘텐츠 폭에 맞춰 정렬 또는 풀폭.

## 단계별 실행 (위험 낮은 순, 각 단계 빌드+육안 검증)

**Phase 1 — 골격 전환 (핵심)**
- ClientLayout 통 제거, GlobalTopNav 신설, CommonNav lg:hidden.
- 콘텐츠 폭 토큰 추가.
- 이 단계에서 PC는 "상단 네비 + 풀폭(아직 정리 안 된) 콘텐츠"가 된다.
  검증: 모바일이 그대로인지 최우선 확인.

**Phase 2 — 콘텐츠 폭 정리**
- 각 페이지 루트를 content-narrow / content-wide 토큰으로 감싼다.
  (PageContainer를 narrow/wide 변형으로 확장.)
- 상세/폼 = narrow, 리스트/홈/마이 = wide.

**Phase 3 — PC 레이아웃 재배치 (화면별)**
- 게시판 리스트: PC 2~3열 그리드 (이미 2열 적용됨 → wide 폭에서 열 수 재검토).
- 홈: 섹션 grid 재배치.
- 상세: 본문 + 사이드(목록/추천) 2단 (선택).
- 마이페이지: 프로필 + 콘텐츠 2단 (이미 일부 적용).

**Phase 4 — fixed 헤더 PC 정리**
- 페이지별 헤더 lg:hidden, GlobalTopNav로 일원화.

## 모바일 불변 보장 (최우선 제약)

- 모든 변경은 `lg:` 분기 또는 "모바일에서 동일하게 보이는" 형태로만.
- 각 Phase 후 모바일 폭(375~640px)에서 **레이아웃이 이전과 동일한지** 육안 검증.
- 통 제거로 바깥 회색 배경이 사라지는데, 모바일은 원래 통이 화면을 꽉 채웠으므로
  체감 변화 없음(확인 필요).

## 리스크

- **높음**. 전역 레이아웃 구조 변경이라 모든 화면에 영향. fixed 요소 정렬,
  헤더 이원화, 폭 토큰 전환이 얽힘.
- 완화: Phase 단위로 쪼개고 각 단계 빌드+육안 검증. 모바일 회귀를 매 단계 확인.
- 커밋은 Phase 단위로. 문제 시 해당 Phase만 되돌림.

## 검증

- 각 Phase: `prettier → npm run build` 통과.
- localhost:3000에서 사용자가 PC/모바일 폭 육안 검증 (내부망이라 gstack 불가).
- 모바일 회귀 0 확인이 배포 전제 조건.
